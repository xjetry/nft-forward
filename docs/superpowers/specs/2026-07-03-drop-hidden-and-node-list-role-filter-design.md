# 删除节点隐藏属性 + 节点列表角色过滤与过滤保持

日期：2026-07-03

## 背景

节点的分类收敛为两个维度：用途（roles：入口 / 中间层）和类型（单点 / 组合）。
「隐藏」是历史上的第三个展示维度——纯前端折叠（不影响转发/同步/授权），与
角色体系并存后语义重叠、心智负担大，决定彻底删除。

同时节点列表需要按用途筛选，且过滤条件要在「进详情 → 返回列表」后保持。

## 设计

### ① 彻底删除 hidden

- 迁移 `0036_drop_node_hidden.sql`：`ALTER TABLE nodes DROP COLUMN hidden`。
  之前被隐藏的节点及其规则会重新出现在管理端列表——这是预期行为。
- 三处 scan 锁步同删：`nodeCols` + `scanNode` + grants.go `ListNodesForUser`。
- 删除 `ToggleNodeHidden`、`POST /nodes/{id}/hidden` 路由与 handler。
- 前端删除：节点列表的「仅展示可见节点」勾选框/隐藏计数/眼睛按钮/已隐藏
  badge、节点详情的隐藏按钮与 badge、规则列表对隐藏节点规则的硬过滤与
  节点筛选候选过滤、拖拽排序对隐藏节点的补位逻辑。

### ② 节点列表：角色过滤 + 过滤保持

- 布局：类型（单点/组合）保持 tab 硬分类；用途做可叠加的筛选钮
  「角色 [入口] [中间层]」，不选 = 全部，多选 = 需兼具（收窄语义）。
- 过滤链：tab → 角色掩码 → 搜索词；任一过滤生效即禁用拖拽调序
  （saveOrder 以可见列表为全量重建顺序，子集视图会丢节点）。
- 持久化：搜索词与角色掩码存 sessionStorage（进详情返回后保持，
  不跨会话——隔天还带着旧过滤会让列表莫名缺节点）；tab 沿用 localStorage。

### ③ 选择器按用途过滤（核查结论，无需改动）

- 规则表单入口选择器已按入口角色过滤（`RuleFormModal.jsx` entryNodes，
  roles bit0），线路层级联已按中间层角色过滤（roles bit1）并有服务端
  双重校验（入口须有 entry 角色、via 须有 via 角色且绑定到上游）。
- 用户详情授权节点选择器本就展示全部节点（仅排除已授权）。
- `Select` 组件 tabs 模式的搜索本就是「先按当前 tab 的组过滤、再对
  label 做大小写不敏感子串匹配」。

## 测试

- 既有 Go 全量测试通过（hidden 无测试引用，删除面干净）。
- 手工验证：带 hidden=1 数据的库经迁移后列正常删除、节点重新可见；
  角色筛选、过滤保持、规则列表、节点详情均正常。
