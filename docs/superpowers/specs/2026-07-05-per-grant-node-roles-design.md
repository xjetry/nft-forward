# 设计：授权级节点角色掩码

## 背景与问题

节点角色是位掩码（`nodes.roles`：`entry`=可作规则入口，`via`=可挂在上游后作中间层）。角色是节点的**全局**属性：一旦某中间层节点打上 `entry`，所有被授权它的用户都能拿它当规则起点，无法只对部分用户开放。

存在一类真实场景无法表达：外部用户自有前置（系统外），只想把系统内某个中间层节点当作「对端」——即让流量从自己的前置打到该中间层节点的监听端口，由它作为链的起点往后转发。物理上每一跳同构（`节点 relay_host:listen_port → target`），入口跳与中间层段首跳在下发上没有区别，所以这只是建模限制：中间层节点在模型里永远是「挂在上游之后的段」，不能被选作起点；即便把它授权给用户，用户的入口选择器也不列它、服务端入口校验也拒它。

给该节点打全局 `entry` 能跑通，但粒度太粗：它会对所有被授权者都变成可选入口，而管理员只想对「自带前置」的少数用户开放。

## 目标

- 「用户能把某节点当作什么用」可按每一条授权独立决定，与节点的全局角色解耦。
- 中间层节点保持其全局中间层身份不变，默认最小权限；只对被明确授予入口能力的授权开放起点。
- 不触碰计费口径、不破坏既有授权语义、改动集中。

## 非目标

- 不为系统外的前置建实体、不建其可达性/计费模型。
- 不做入口监听端口的源 IP 白名单（裸端口开放是既有状况，本设计不涉及，仅记为已知点）。
- 存量授权已用量不回算。

## 概念模型

- **节点掩码 `nodes.roles`**：节点的默认角色，语义不变。表达「这个节点默认被当作什么」。
- **授权掩码 `user_nodes.roles`**：这一条授权把节点当作什么用。与节点掩码**对等且独立**——同一取值空间（`entry`/`via` 的任意非空组合），谁都不是谁的子集或超集。授权掩码可加可减，覆盖后完全独立于节点掩码。
- **哨兵与继承**：授权掩码 `0` 表示「继承节点掩码」；非 `0` 表示「覆盖为该值」。选用 `0` 作哨兵与 `nodes.roles != 0` 的既有约束一致（有效角色为空集无意义，故 `0` 不会是合法的覆盖值）。
- **有效角色**（唯一新增判定原语）：`effectiveRoles = grant.roles == 0 ? node.roles : grant.roles`。

于是中间层节点 M（节点掩码纯 `via`）：授权给甲覆盖成 `entry`（或 `entry|via`），甲的有效角色含 `entry`，可选 M 当起点；授权给乙继承默认，乙有效角色为 `via`，只能拿 M 当中间层。M 的全局身份和默认权限都不变。

## 判定归属（不变量）

角色判定按「是否与具体用户有关」二分，这条边界是本设计的核心不变量：

- **per-user 判定读有效角色**：某用户能否把节点当入口、当中间层——my 路径的入口候选与校验、via 候选与校验。
- **节点能力/拓扑判定读节点掩码**：绑定图的 downstream 必须具 `via`（`internal/server/api.go` 绑定校验）、管理端节点角色管理。授权掩码不得篡改管理员定义的绑定拓扑——绑定图是「节点之间能怎么接」的客观拓扑，授权掩码只表达「某用户能把节点当什么用」。

## 数据模型

- `user_nodes.roles INTEGER NOT NULL DEFAULT 0`。迁移将所有存量授权置 `0`（继承），行为零变化。
- 加列必须同步所有读 `user_nodes` 的位置，漏一处会静默错读授权掩码：`UserNode` 结构、`scanUserNode`、`ListNodesForUser` 的 inline SELECT、`GetNodeGrant`、`ListUsersForNode`、`ListAllGrants`。
- 覆盖值合法域 `{entry, via, entry|via}`；写入时校验 `roles &^ (entry|via) == 0`，`0` 合法（继承）。

## 服务端

- `hopsForChain` 现按节点掩码校验入口（要求 `entry`）与每个 via（要求 `via`）。改为按**有效角色**校验：判定所需的 grant 在 `apiMyCreateRule` 已取到（入口 grant 与各 via grant）。有效角色的合并要么在 handler 层算好传入 `hopsForChain`，要么把角色校验上提到 handler；`hopsForChain` 仍独占绑定边与结构校验。
- **入口能力判定**：my 路径入口节点的有效角色须含 `entry`；否则拒绝。
- **via 能力判定**：my 路径每个 via 节点的有效角色须含 `via`；绑定边（`GetNodeBinding`）与物理去重校验不变。
- **admin 派生路径**（`node_id + via`）：按规则 owner 对该节点的 grant 有效角色判定，与 my 路径同一原语；owner 无该节点 grant 时回退节点掩码。admin 的 explicit-hops 逃生舱语义不变（本就不校验角色与绑定，仅守 `no_direct_exit`）。

## 计费

零改动。有效角色只决定「能不能选」，不参与选中之后的记账。中间层节点被当作入口时它处于规则 position 0（`rules.node_id` 即该节点），既有「入口段按节点 `rate_multiplier` 计一次全局用量」的逻辑原样生效；它作为别人链路的 via 时仍走 via 口径。口径由节点在链里的位置决定，与授权掩码无关。

## 前端与授权

- **my 节点列表 API**：对每个授权节点返回**有效角色**（服务端合并 `grant.roles || node.roles`）。my 上下文里节点的 `roles` 字段即有效角色，`RuleFormModal` 的入口/via 候选过滤逻辑无需改动。admin 上下文的节点 `roles` 仍是节点掩码。
- **授权掩码写入**：通过专用 per-node 设置端点 `POST /users/{id}/nodes/{nodeID}/roles`（与 `max-forwards`/`quota`/`rate-limit` 同一模式）调整，`0`=继承。`GrantNode`（含批量与初始授权）不变——新建授权走列默认 `0` 继承节点掩码，既有与批量授权路径无需改动，也不会被静默改写。
- **授权编辑/展示 UI**：`users/Detail.jsx` 的已授权表格加「用途」下拉（跟随节点 / 仅入口 / 仅中间层 / 入口+中间层）调上述端点，一眼可见哪些授权被特意放开或收窄；`nodes/Detail.jsx` 的已授权用户表只读展示同一用途。批量粘贴授权（`PasteGrantsModal`）默认继承，不引入掩码列。

## 迁移与兼容

- `user_nodes.roles` 默认 `0`，全部继承，既有授权与入口行为不变。
- 入口仍由节点掩码驱动（继承），授权掩码只在被显式覆盖时生效，属于叠加式的定向放开/收窄。
- 普通与批量授权不设掩码，默认继承；掩码仅经专用 per-node 端点显式改写。

## 测试要点

- 有效角色合并：`grant.roles==0` 取节点掩码、非 `0` 取覆盖值；覆盖可超出节点掩码（节点 `via`、授权覆盖 `entry` → 有效 `entry`），也可收窄（节点 `entry|via`、授权覆盖 `via`）。
- my 路径：中间层节点授权覆盖含 `entry` 后可作起点、两种终点（直落目标 / 接下游 via 链）均成链；未覆盖的授权仍拒作入口；授权覆盖去掉 `via` 后不能再作中间层。
- 拓扑不变量：授权掩码不影响绑定图 downstream 的 `via` 能力校验；被授权用户不能凭授权掩码绕过缺失的绑定边。
- admin 派生路径：按 owner 有效角色判定，无 owner grant 回退节点掩码；explicit-hops 逃生舱不受影响。
- 计费：中间层节点作入口时按其 `rate_multiplier` 计一次全局用量；作 via 时口径不变。
- 加列对齐：所有 `user_nodes` scan 点同步，授权掩码读写一致。
- 兼容：存量授权默认继承（掩码 `0`）；掩码仅经专用 per-node 端点改写，普通/批量授权路径不受影响。

## 设计取舍与理由

- **授权掩码与节点掩码对等独立**：把「节点默认是什么」与「某用户能把它当什么」拆成两个自由度，用一个继承哨兵连接。既能对少数用户放开中间层作入口（白名单式扩展），也能对个别用户收窄公开入口节点（黑名单式裁剪），而不必为双重身份复制节点或改动全局角色。
- **默认继承、最小暴露**：默认 `0` 继承使存量与新授权行为都不变，定向能力是纯叠加，避免一次迁移改变既有用户的可用面。
- **判定按 per-user / 拓扑二分**：授权掩码只作用于「用户视角的可用性」，绑定图等客观拓扑始终读节点掩码，防止授权层悄悄改变管理员定义的链路结构。
- **计费不动**：计费口径挂在节点在链中的位置上，与「谁被允许怎么用」正交，因此本设计不触碰任何记账路径。
