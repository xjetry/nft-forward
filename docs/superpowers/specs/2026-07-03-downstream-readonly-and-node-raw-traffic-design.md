# 节点详情下游只读展示 + 节点列表原始流量列

日期：2026-07-03

## 背景与问题

两个独立的小改进，均在管理端：

1. **节点详情**：「节点角色」卡目前只有本节点作为下游的「上游绑定」编辑器。
   反向关系（哪些节点把本节点绑定为上游）没有任何呈现，管理员要逐个打开其他
   节点详情才能拼出中间层图的入边。
2. **节点列表的「流量」列不是原始流量**。它是 `SUM(user_nodes.traffic_used_bytes)`，
   即按授权记账的当期用量：虽不乘倍率，但存在三个口径偏差——
   - 单向计费节点只计上行；
   - 组合节点记在逻辑节点授权上，物理子节点不累计；
   - 用户流量周期重置 / 管理员重置会清零。

   管理员无法看到节点真实承载的转发字节数。

## 设计

### ① 已绑定的下游（只读）

- **数据**：复用管理端已有的 `GET /node-bindings` 全量边接口，前端过滤
  `upstream_node_id === 当前节点`。无后端改动。
- **位置**：「节点角色」卡内、上游绑定编辑器之后（无中间层勾选时紧跟角色行），
  独立一个带上边框的小节。**不依赖「中间层」勾选**——入口节点同样可以被下游
  绑定为上游。
- **展示条件**：有 ≥1 条下游边才显示；无下游时该小节完全不出现。
- **内容**：每行 = 节点类型图标 + 下游节点名（链接到该下游详情页）+ 衔接模式
  badge；附说明「绑定关系在对应下游节点的详情页编辑」。
- **失败语义**：加载失败按空处理（纯只读展示，无覆盖风险，不做重试 UI）。
- **节点名来源**：候选节点列表的拉取条件从「勾选中间层」放宽为
  「勾选中间层 或 存在下游边」，入口-无下游节点保持零额外请求。

### ② 节点原始流量

**口径**：物理节点实际转发的字节（上行+下行），来自 agent 上报的增量样本，
不乘倍率、不受单向计费影响、不随任何用户流量重置清零；累计值。
组合节点是逻辑节点、不上报，没有原始流量可言。

- **存储**：新表（迁移 `0035_node_raw_traffic.sql`）：

  ```sql
  CREATE TABLE node_raw_traffic (
      node_id   INTEGER PRIMARY KEY REFERENCES nodes(id) ON DELETE CASCADE,
      raw_bytes INTEGER NOT NULL DEFAULT 0
  );
  ```

  不放 `nodes` 列的原因：`nodeCols`/`scanNode`/grants.go 三处 scan 需锁步；且
  Node JSON 会流向普通用户端点（`ListNodesForUser`），全节点吞吐总量属于管理
  信息，不应顺带暴露。独立表 + map 下发与现有 `node_traffic` 形态一致。
- **累加点**：`hub.applyCounters` 的样本循环里累计 `totalDelta`（在 rule_hop
  匹配之前——规则已删除的样本同样是真实转发字节），批次刷盘事务里 upsert 一次。
- **API**：仅管理端 `GET /nodes` 响应新增 `node_raw_traffic` map（dashboard
  不加，普通用户不可见）。
- **前端**：节点列表新增「原始流量」列，桌面表格与移动卡片两个渲染路径都加，
  可排序；单点/组合按 tab 完全分列，组合 tab 不渲染该列（移动卡片对组合节点
  不显示「原始」项）。切到组合 tab 时清掉 rawtraffic 排序——该列不渲染，排序
  会不可见且无法取消。现有「流量」列保留原样。

## 测试

- Go（server 包，沿用 `traffic_accounting_test.go` 模式）：
  - 双向样本累加 = up+down；单向计费节点 raw 仍计双向（与 billed 口径分离）；
  - 无匹配 rule_hop 的样本仍计入 raw；
  - `ResetAllUserTraffic` 不影响 raw 计数；
  - 删除节点级联删除 raw 行。
- 前端无测试基建：`npm run build` + 人工核对两个渲染路径。

## 不做的事（YAGNI）

- 不做 raw 计数的重置入口；
- 不改 dashboard / 用户侧任何流量展示；
- 不为下游列表加编辑能力（编辑归属下游节点详情页，保持单一编辑入口）。
